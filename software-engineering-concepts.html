Principles & SOLID

Inversion of Control (IoC)

Principle ▼
Don't call us, we'll call you — the framework controls the flow, not your code.
Problem
Your code controls everything: object creation, lifecycle, flow. Tight coupling everywhere. Changing one part means rewriting many others.
Bad Approach
// Your code controls everything
class OrderService {
  process(order) {
    const db = new MySQLDatabase();      // YOU create it
    const mailer = new SmtpMailer();      // YOU create it
    const logger = new FileLogger();      // YOU create it
    db.save(order);
    mailer.send(order.email, "Confirmed");
    logger.log("Order processed");
  }
}
Good Approach
// Framework/container controls creation and wiring
class OrderService {
  constructor(db, mailer, logger) {   // GIVEN to you
    this.db = db;
    this.mailer = mailer;
    this.logger = logger;
  }
  process(order) {
    this.db.save(order);
    this.mailer.send(order.email, "Confirmed");
    this.logger.log("Order processed");
  }
}
Strengths
  • Loose coupling
  • Easy to test with mocks
  • Swappable implementations
Weaknesses
  • Harder to trace flow
  • Container configuration complexity
  • Learning curve
Real-World
Spring Framework, Angular, NestJS — your controllers/services are instantiated and wired by the framework, not by you calling new.
Vue 3 / TS Example
Vue 3 / TS
// Vue's plugin system — Vue controls the lifecycle, not you
const app = createApp(App);
app.use(piniaPlugin);     // Vue calls plugin.install()
app.use(routerPlugin);    // You don't call router.init()
app.use(i18nPlugin);      // Framework orchestrates everything

// provide/inject — parent controls what child receives
// Parent component
const apiService = new ApiService();
provide('api', apiService);  // Parent decides what to provide

// Child component — doesn't create its own dependency
const api = inject('api');   // Given to you, not created by you

Dependency Injection (DI)

Principle ▼
Pass dependencies in from outside instead of creating them inside.
Problem
Classes create their own dependencies, making them impossible to test in isolation or swap implementations.
Bad Approach
class UserController {
  getUser(id) {
    const repo = new PostgresUserRepo(); // Hardcoded!
    return repo.findById(id);
  }
}
// Can't test without a real Postgres instance
Good Approach
class UserController {
  constructor(private userRepo: UserRepository) {}

  getUser(id) {
    return this.userRepo.findById(id);
  }
}
// Test: new UserController(mockRepo)
// Prod: new UserController(new PostgresUserRepo())
Strengths
  • Testability (inject mocks)
  • Flexibility (swap DB, cache, etc.)
  • Explicit dependencies
Weaknesses
  • More boilerplate / wiring
  • Can hide complexity in container config
Real-World
Any backend framework — inject a PaymentGateway interface so you can swap Stripe for a mock in tests or switch to Adyen in production.
Vue 3 / TS Example
Vue 3 / TS
// Typed provide/inject with InjectionKey
interface AuthService { login(token: string): Promise<void>; logout(): void; }
class RealAuthService implements AuthService {
  async login(token: string) { /* call API */ }
  logout() { /* clear session */ }
}

const AuthServiceKey: InjectionKey<AuthService> = Symbol('auth');

// Provider component
provide(AuthServiceKey, new RealAuthService());

// Consumer component — depends on abstraction
const auth = inject(AuthServiceKey)!;
auth.login('telegram-init-data'); // uses the injected service

// Composable accepting deps as params (testable!)
interface WalletApi { getBalance(): Promise<string>; }
const defaultWalletApi: WalletApi = { getBalance: () => api.get('/balance') };

function useWallet(walletApi: WalletApi = defaultWalletApi) {
  const balance = ref('0');
  const fetchBalance = async () => {
    balance.value = await walletApi.getBalance();
  };
  return { balance, fetchBalance };
}
// Test: useWallet(mockApi)  Prod: useWallet()

SRP — Single Responsibility Principle

SOLID ▼
A class should have only one reason to change.
Problem
A god-class does everything — parsing, validation, DB access, email — so any change risks breaking unrelated features.
Bad Approach
class Invoice {
  calculate() { /* business logic */ }
  saveToDb() { /* persistence */ }
  generatePdf() { /* presentation */ }
  sendEmail() { /* notification */ }
}
Good Approach
class Invoice { calculate() { /* only business logic */ } }
class InvoiceRepository { save(inv) { /* only persistence */ } }
class InvoicePdfGenerator { generate(inv) { /* only PDF */ } }
class InvoiceNotifier { notify(inv) { /* only email */ } }
Strengths
  • Changes are isolated
  • Easier to test each piece
  • Better team parallelism
Weaknesses
  • More classes to manage
  • "One reason to change" is subjective
Real-World
Microservices — each service owns one domain concern. Changing billing logic doesn't require redeploying the notification service.
Vue 3 / TS Example
Vue 3 / TS
// BAD — one composable doing everything
function useUser() {
  // fetching + validation + formatting + caching... 🤮
}

// GOOD — each composable has one responsibility
function useSearch() { /* only search state & query */ }
function useDebounce(value, ms) { /* only debounce logic */ }
function useUserApi() { /* only API calls */ }

// Component split: Container vs Presentational
// WalletContainer.vue — data fetching, state management
// WalletAssetsList.vue — pure presentation, receives props
// walletService.ts — API calls only
// useWalletStore.ts — state management only

OCP — Open/Closed Principle

SOLID ▼
Open for extension, closed for modification — add behavior without editing existing code.
Bad Approach
function calculateDiscount(type, amount) {
  if (type === "regular") return amount * 0.05;
  if (type === "premium") return amount * 0.1;
  if (type === "vip") return amount * 0.2;
  // Adding new type? Edit this function every time!
}
Good Approach
interface DiscountStrategy { calculate(amount: number): number; }

class RegularDiscount implements DiscountStrategy {
  calculate(amount) { return amount * 0.05; }
}
class VipDiscount implements DiscountStrategy {
  calculate(amount) { return amount * 0.2; }
}
// New type? Just add a new class. No existing code modified.
Strengths
  • New features without regression risk
  • Stable, tested core
Weaknesses
  • Requires upfront abstraction design
  • Over-engineering if requirements are stable
Real-World
Plugin systems (Webpack, VS Code) — core is closed; plugins extend behavior without modifying the core.
Vue 3 / TS Example
Vue 3 / TS
// Dynamic component — extend without modifying
<component :is="transactionIcons[tx.type]" />

const transactionIcons: Record<string, Component> = {
  send: TxSendIcon,
  receive: TxReceiveIcon,
  // Add new type here — no existing code changes
  swap: TxSwapIcon,
};

// Composable accepting strategy function
interface Asset { symbol: string; name: string; price: number; }
const assets = ref<Asset[]>([]);

function useSortedList<T>(items: Ref<T[]>, compareFn: (a: T, b: T) => number) {
  return computed(() => [...items.value].sort(compareFn));
}
// Open for extension — pass any sort strategy
const byPrice = useSortedList(assets, (a, b) => b.price - a.price);
const byName = useSortedList(assets, (a, b) => a.name.localeCompare(b.name));

LSP — Liskov Substitution Principle

SOLID ▼
Subtypes must be substitutable for their base types without breaking correctness.
Bad Approach
class Rectangle {
  setWidth(w) { this.w = w; }
  setHeight(h) { this.h = h; }
  area() { return this.w * this.h; }
}
class Square extends Rectangle {
  setWidth(w) { this.w = w; this.h = w; } // Breaks expectations!
  setHeight(h) { this.w = h; this.h = h; }
}
// rect.setWidth(5); rect.setHeight(3); rect.area() → expects 15
// With Square: area() → 9. Surprise!
Good Approach
interface Shape { area(): number; }
class Rectangle implements Shape { /* width × height */ }
class Square implements Shape { /* side × side */ }
// No inheritance, no broken substitution
Strengths
  • Polymorphism works correctly
  • No surprises for callers
Weaknesses
  • Can limit inheritance hierarchies
  • Requires careful contract design
Real-World
Collection APIs — List, Set, Map all implement Iterable. Any code accepting Iterable works with all of them.
Vue 3 / TS Example
Vue 3 / TS
// Props contract — IconButton must honor BaseButton's interface
// BaseButton.vue
defineProps<{ label: string; disabled?: boolean; onClick: () => void }>();

// IconButton.vue — extends BaseButton, substitutable everywhere
defineProps<{ label: string; disabled?: boolean; onClick: () => void; icon: string }>();

// Parent can swap BaseButton → IconButton without breaking
<BaseButton v-bind="buttonProps" />
<IconButton v-bind="buttonProps" icon="star" />  // ✓ Works!

// Consistent emit interfaces across related components
// All list components emit: select(item), remove(item)
// AssetsList, CurrenciesList, TokenList — all substitutable

ISP — Interface Segregation Principle

SOLID ▼
No client should be forced to depend on methods it doesn't use.
Bad Approach
interface Worker {
  code(): void;
  design(): void;
  test(): void;
  managePeople(): void;
}
// Junior dev must implement managePeople()? Throws NotImplemented?
Good Approach
interface Coder { code(): void; }
interface Designer { design(): void; }
interface Manager { managePeople(): void; }

class FullStackDev implements Coder, Designer { ... }
class TeamLead implements Coder, Manager { ... }
Strengths
  • Clients only see what they need
  • Less coupling, easier refactoring
Weaknesses
  • Many small interfaces to manage
  • Can fragment cohesive concepts
Real-World
Java's Readable/Writable/Closeable — a read-only stream doesn't need write methods.
Vue 3 / TS Example
Vue 3 / TS
// BAD — one huge composable forces consumers to depend on everything
function useAuthAll() {
  const user = ref<string | null>(null);
  const roles = ref<string[]>([]);
  const login = () => { /* ... */ };
  const logout = () => { /* ... */ };
  const adminPanel = ref(false);
  const auditLog = ref<string[]>([]);
  return { login, logout, user, roles, adminPanel, auditLog };
}

// GOOD — split into focused composables
function useAuth() {
  const user = ref<string | null>(null);
  const login = () => { /* ... */ };
  const logout = () => { /* ... */ };
  return { login, logout, user };
}
function useAuthAdmin() {
  const roles = ref<string[]>([]);
  const adminPanel = ref(false);
  return { roles, adminPanel };
}

// TypeScript: Pick only what you need
interface User { id: string; name: string; avatar: string; email: string; }
type UserCardProps = Pick<User, 'name' | 'avatar'>;
// Instead of passing the entire User object to a display card

DIP — Dependency Inversion Principle

SOLID ▼
High-level modules depend on abstractions, not concrete implementations.
Bad Approach
class OrderService {
  private repo = new MySQLOrderRepo(); // High-level depends on low-level
}
// Switching to MongoDB? Rewrite OrderService.
Good Approach
interface OrderRepository { save(o: Order): void; }

class OrderService {
  constructor(private repo: OrderRepository) {}
}
// Both high-level and low-level depend on the abstraction
Real-World
Clean Architecture layers — domain layer defines interfaces, infrastructure layer implements them. Domain never imports infrastructure.
Vue 3 / TS Example
Vue 3 / TS
// Define interface (abstraction)
interface PaymentGateway {
  charge(amount: string): Promise<Receipt>;
}

// InjectionKey typed to interface, not implementation
const PaymentKey: InjectionKey<PaymentGateway> = Symbol('payment');

// High-level composable depends on abstraction
function useCheckout() {
  const gateway = inject(PaymentKey)!; // interface, not StripeService
  const pay = (amount: string) => gateway.charge(amount);
  return { pay };
}

// In app setup — provide concrete implementation
provide(PaymentKey, new StripeGateway());
// In tests — provide mock
provide(PaymentKey, mockGateway);

GRASP Patterns

GRASP — General Responsibility Assignment

Principle ▼
Nine principles for deciding which class gets which responsibility.
The 9 Patterns
1. Information Expert — Assign responsibility to the class with the most info needed to fulfill it.
2. Creator — B creates A if B contains/aggregates A, closely uses A, or has init data for A.
3. Controller — A non-UI object handles system events (use-case controller or facade controller).
4. Low Coupling — Minimize dependencies between classes for resilience to change.
5. High Cohesion — Keep related responsibilities together; a class does one focused thing well.
6. Polymorphism — Use polymorphic methods instead of conditionals on type.
7. Pure Fabrication — Invent a class that doesn't represent a domain concept to achieve low coupling/high cohesion.
8. Indirection — Assign responsibility to an intermediate object to decouple two components.
9. Protected Variations — Wrap instability behind a stable interface.
Example: Information Expert
// BAD: External code calculates order total
function calcTotal(order) {
  return order.items.reduce((sum, i) => sum + i.price * i.qty, 0);
}
// GOOD: Order has the info, so Order calculates it
class Order {
  getTotal() {
    return this.items.reduce((sum, i) => sum + i.price * i.qty, 0);
  }
}
Real-World
Pure Fabrication example: OrderRepository is not a domain concept but is fabricated to separate persistence from domain logic, achieving low coupling.
Vue 3 / TS Example
Vue 3 / TS
// Information Expert — component with data computes derived state
// WalletView.vue has assets → it computes total balance

  assets.value.reduce((sum, a) => Big(sum).plus(a.balanceUsd).toString(), '0')

  return { assets, toggleVisibility };

GoF Design Patterns

Builder

Creational ▼
Construct complex objects step-by-step, separating construction from representation.
Bad Approach
// Constructor with 10 parameters — unreadable, error-prone
const query = new Query("users", ["name"], null, "age > 18",
  null, "name", "ASC", 10, 0, true);
Good Approach
const query = new QueryBuilder("users")
  .select("name")
  .where("age > 18")
  .orderBy("name", "ASC")
  .limit(10)
  .build();
Strengths
  • Readable, self-documenting
  • Validates at build time
  • Immutable result objects
Weaknesses
  • More code for simple objects
  • Duplicated field definitions
Real-World
SQL query builders (Knex, TypeORM QueryBuilder), HTTP request builders, StringBuilder in Java.
Vue 3 / TS Example
Vue 3 / TS
// Fluent query params builder
class QueryBuilder {
  private params = new URLSearchParams();
  page(n: number) { this.params.set('page', String(n)); return this; }
  limit(n: number) { this.params.set('limit', String(n)); return this; }
  filter(k: string, v: string) { this.params.set(k, v); return this; }
  build() { return this.params.toString(); }
}

const query = new QueryBuilder()
  .page(1).limit(20).filter('status', 'active').build();

// TanStack Query options builder pattern
const { data } = useQuery({
  queryKey: ['walletAssets', currency],
  queryFn: () => walletApi.getAssets(currency),
  staleTime: 30_000,
  retry: 3,
});

Adapter

Structural ▼
Convert one interface into another that clients expect — a plug converter for code.
Bad Approach
// Client code has to deal with every payment provider's API
if (provider === "stripe") stripe.charges.create({amount});
else if (provider === "paypal") paypal.payment.execute({total: amount});
Good Approach
interface PaymentGateway { charge(amount: number): Receipt; }

class StripeAdapter implements PaymentGateway {
  charge(amount) { return stripe.charges.create({amount}); }
}
class PayPalAdapter implements PaymentGateway {
  charge(amount) { return paypal.payment.execute({total: amount}); }
}
// Client just calls gateway.charge(100) — doesn't care which provider
Real-World
ORMs adapting to different databases, logging facades (SLF4J wrapping Log4j/Logback), API version migration layers.
Vue 3 / TS Example
Vue 3 / TS
// API response adapter: snake_case → camelCase
interface ApiTransaction {
  tx_id: string; created_at: string; amount_usd: string;
}
interface Transaction {
  txId: string; createdAt: string; amountUsd: string;
}

function adaptTransaction(raw: ApiTransaction): Transaction {
  return {
    txId: raw.tx_id,
    createdAt: raw.created_at,
    amountUsd: raw.amount_usd,

}

// Third-party lib wrapper composable
// Wraps Telegram WebApp SDK into Vue-friendly reactive API
function useTelegram() {
  const user = ref(window.Telegram?.WebApp?.initDataUnsafe?.user);
  const close = () => window.Telegram?.WebApp?.close();
  return { user, close };
}

Facade

Structural ▼
Provide a simplified interface to a complex subsystem.
Bad Approach
// Client must orchestrate 5 subsystems to place an order
inventory.reserve(items);
const tax = taxService.calculate(items, address);
const charge = payment.charge(total + tax);
shipping.createLabel(address, items);
notification.sendConfirmation(email, orderId);
Good Approach
class OrderFacade {
  placeOrder(items, address, email) {
    inventory.reserve(items);
    const tax = taxService.calculate(items, address);
    payment.charge(total + tax);
    shipping.createLabel(address, items);
    notification.sendConfirmation(email);
  }
}
// Client: orderFacade.placeOrder(items, addr, email);
Real-World
jQuery was a facade over inconsistent browser DOM APIs. AWS SDK high-level clients simplify multi-step S3 uploads.
Vue 3 / TS Example
Vue 3 / TS
// Complex subsystem: 3 stores + API + router
// Facade composable hides all complexity
function useCheckout() {
  const cart = useCartStore();
  const wallet = useWalletStore();
  const auth = useAuthStore();
  const router = useRouter();

  const checkout = async () => {
    if (!auth.isLoggedIn) return router.push('/login');
    await walletApi.charge(cart.total);
    wallet.refreshBalance();
    cart.clear();
    router.push('/success');
  };

  return { checkout }; // Simple interface!
}

// Component just calls: const { checkout } = useCheckout();

Proxy

Structural ▼
A surrogate that controls access to the real object (lazy load, cache, auth).
Good Approach
class CachingProxy {
  constructor(private realService: DataService) {}
  cache = new Map();

  getData(key) {
    if (!this.cache.has(key)) {
      this.cache.set(key, this.realService.getData(key));
    }
    return this.cache.get(key);
  }
}
// Same interface as realService — client doesn't know it's cached
Strengths
  • Transparent to clients
  • Lazy init, caching, access control
Weaknesses
  • Adds indirection
  • Can mask performance issues
Real-World
ORM lazy loading (Hibernate proxies), JS Proxy object, reverse proxy (Nginx), virtual DOM.
Vue 3 / TS Example
Vue 3 / TS
// Vue 3 reactivity IS the Proxy pattern!
const state = reactive({ count: 0 });
// Under the hood: new Proxy(target, {
//   get(target, key) { track(target, key); return target[key]; },
//   set(target, key, value) { target[key] = value; trigger(target, key); }
// })

// computed() is a caching proxy
const first = ref('John');
const last = ref('Doe');
const fullName = computed(() => `${first.value} ${last.value}`);
// Only recomputes when dependencies change — lazy + cached

// watchEffect — lazy evaluation proxy
watchEffect(() => {
  console.log(state.count); // Auto-tracks, re-runs on change
});

Decorator

Structural ▼
Wrap an object to add behavior dynamically without subclassing.
Bad Approach
// Explosion of subclasses
class EncryptedCompressedLogger extends Logger { ... }
class EncryptedLogger extends Logger { ... }
class CompressedLogger extends Logger { ... }
// N features → 2^N classes!
Good Approach
class EncryptionDecorator {
  constructor(private inner: DataSink) {}
  write(data) { this.inner.write(encrypt(data)); }
}
class CompressionDecorator {
  constructor(private inner: DataSink) {}
  write(data) { this.inner.write(compress(data)); }
}
// Compose: new Encryption(new Compression(new FileWriter()))
Real-World
Java I/O Streams: new BufferedReader(new InputStreamReader(new FileInputStream(f))). Express/Koa middleware chains.
Vue 3 / TS Example
Vue 3 / TS
// Wrapping a composable to add logging
function withLogging<T>(composable: () => T, name: string): () => T {
  return () => {
    console.log(`[${name}] initialized`);
    return composable();
  };
}
function useWallet() { return { balance: ref('0') }; }
const useWalletWithLogs = withLogging(useWallet, 'Wallet');

// Axios interceptors as decorators
const token = useCookies().get('auth_token');
api.interceptors.request.use(config => {
  config.headers.Authorization = `Bearer ${token}`; // Decorate request
  return config;
});
api.interceptors.response.use(
  res => res,
  err => refreshOrInitializeToken().then(() => api(err.config)) // Decorate error handling
);

Composite

Structural ▼
Treat individual objects and compositions uniformly via a tree structure.
Good Approach
interface FileSystemNode { getSize(): number; }

class File implements FileSystemNode {
  getSize() { return this.bytes; }
}
class Directory implements FileSystemNode {
  children: FileSystemNode[] = [];
  getSize() { return this.children.reduce((s, c) => s + c.getSize(), 0); }
}
// directory.getSize() recursively sums all files — uniform interface
Real-World
DOM tree, React component tree, file systems, organization charts, UI layout containers.
Vue 3 / TS Example
Vue 3 / TS
// Recursive component tree — nested menu
// MenuItem.vue
<template>
  <li>
    {{ item.label }}
    <ul v-if="item.children?.length">
      <MenuItem v-for="child in item.children"
        :key="child.id" :item="child" />
    </ul>
  </li>
</template>

// Slot composition — uniform interface for leaf & composite
<Card>
  <CardHeader>Title</CardHeader>
  <CardBody>
    <Card><!-- Nested card, same interface --></Card>
  </CardBody>
</Card>

Bridge

Structural ▼
Decouple abstraction from implementation so both can vary independently.
Bad Approach
// 2 shapes × 3 renderers = 6 classes

Domain-Driven Design (DDD)

Architecture ▼
Model software around the business domain using ubiquitous language and bounded contexts.
Key Concepts
Ubiquitous Language — developers and domain experts use the same terms in code and conversation.

Entities — objects with identity (User, Order).
Value Objects — immutable, no identity (Money, Address).
Aggregates — cluster of entities with a root that ensures consistency.
Domain Events — something meaningful that happened (OrderPlaced, PaymentReceived).
Repositories — persist/retrieve aggregates.

Clean Architecture

Architecture ▼
Dependencies point inward — business rules know nothing about frameworks or DB.
Layers (inside → out)
1. Entities — enterprise-wide business rules (pure domain objects).
2. Use Cases — application-specific business rules (orchestrate entities).
3. Interface Adapters — controllers, presenters, gateways (convert data between layers).
4. Frameworks & Drivers — DB, web framework, UI (outermost, most volatile).
Key Rule
// Inner layers NEVER import from outer layers
// Use Case defines interface:

  execute(id) { return this.repo.findById(id); }
}
Strengths
  • Framework-independent
  • Highly testable
  • Business rules protected from change
Weaknesses
  • Lots of interfaces & mapping
  • Overkill for simple CRUD apps

CQRS

Architecture ▼
Separate read models from write models for independent scaling and optimization.
Bad Approach
// Same model for reads and writes

  create(p) { db.products.insert(p); }
  getById(id) { return db.products.findById(id); }
  search(q) { return db.products.find(q); } // Slow! Joins everywhere
}
// Write-optimized schema conflicts with read-optimized queries
Good Approach
// Write side — normalized, validates business rules
class CreateProductHandler {
  handle(cmd) { writeDb.products.insert(cmd); eventBus.emit("ProductCreated"); }
}

// Read side — denormalized, optimized for queries
class ProductReadModel {
  onProductCreated(e) { readDb.productViews.upsert(flatten(e)); }
}
class ProductQuery {
  search(q) { return readDb.productViews.find(q); } // Fast! Pre-joined
}
Strengths
  • Independent scaling of reads/writes
  • Optimized query models
  • Simpler read logic
Weaknesses
  • Eventual consistency between models
  • Increased complexity
  • Data sync infrastructure needed
Real-World
E-commerce search — writes go to PostgreSQL (normalized), reads go to Elasticsearch (denormalized). Twitter timeline — fan-out on write to pre-computed feeds.

SAGA Pattern

Distributed ▼
Manage distributed transactions via a sequence of local transactions with compensating actions.
Problem
Distributed systems can't use traditional 2-phase commit (too slow, availability killer). How do you ensure consistency across services?
Good Approach
// Saga: sequence of local transactions + compensations
// Book Trip Saga:
1. Reserve flight    → compensation: cancelFlight()
2. Reserve hotel     → compensation: cancelHotel()
3. Charge payment    → compensation: refund()

// If step 3 fails:
// → cancelHotel() → cancelFlight() (reverse order)

class BookTripSaga {
  steps = [
    { action: reserveFlight, compensate: cancelFlight },
    { action: reserveHotel,  compensate: cancelHotel },
    { action: chargePayment, compensate: refund },
  ];
  async execute() {
    const completed = [];
    for (const step of this.steps) {
      try { await step.action(); completed.push(step); }
      catch { for (const s of completed.reverse()) await s.compensate(); }
    }
  }
}
Strengths
  • No distributed locks
  • Each service stays autonomous
  • Fault recovery built in
Weaknesses
  • Compensations can fail too
  • Eventual consistency (not immediate)
  • Complex to debug
Real-World

Unit of Work

Good Approach
class UnitOfWork {
  private dirty = [];  private new$ = [];  private removed = [];

  markDirty(entity) { this.dirty.push(entity); }
  markNew(entity) { this.new$.push(entity); }

  async commit() {
    const tx = await db.beginTransaction();
    try {
      for (const e of this.new$) await tx.insert(e);
      for (const e of this.dirty) await tx.update(e);
      await tx.commit();
    } catch { await tx.rollback(); }
  }
}

Big O Notation

Theory ▼
Upper bound on growth rate — how algorithm time/space scales with input size.
Common Complexities

      if (arr[i] === arr[j]) return true;
}
// O(n) — use a Set
function hasDupes(arr) {
  const seen = new Set();
  for (const x of arr) {
    if (seen.has(x)) return true;
    seen.add(x);
  }
}

Amortized Complexity

Theory ▼
Average cost per operation over a worst-case sequence — occasional expensive ops are OK.
Classic Example: Dynamic Array
// ArrayList.add() is O(1) amortized:
// Most appends: O(1) — just write to next slot
// Occasionally: O(n) — array full, copy everything to 2× bigger array
//
// After n insertions: n cheap + log(n) expensive
// Total work ≈ n + n/2 + n/4 + ... ≈ 2n → O(n) total → O(1) per op

// Other examples:
// - Hash table rehashing: O(1) amortized insert
// - Splay tree: O(log n) amortized search
Key Insight
  • Don't over-optimize for rare worst cases
  • Bulk operations average out spikes
Watch Out
  • Not suitable for real-time systems (individual ops can spike)
  • Different from average case (amortized is worst-case guaranteed)

Distributed Systems

CAP Theorem

Distributed ▼
Pick two of three: Consistency, Availability, Partition tolerance — you can't have all three.
The Three Properties
C — Consistency: Every read receives the most recent write.
A — Availability: Every request receives a response (not an error).
P — Partition Tolerance: System continues despite network splits between nodes.
Trade-offs
// CP (Consistency + Partition tolerance): refuse requests during partitions
// → MongoDB (default), HBase, Zookeeper
// Use when: financial transactions, inventory counts

// AP (Availability + Partition tolerance): serve stale data during partitions
// → Cassandra, DynamoDB, CouchDB
// Use when: social media feeds, shopping carts

// CA (Consistency + Availability): impossible in distributed systems
// → Only single-node databases (traditional RDBMS)

ACID

Database ▼
Atomicity, Consistency, Isolation, Durability — guarantees for reliable DB transactions.
Properties
// A — Atomicity: all or nothing
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT; // Both succeed or both fail

// C — Consistency: DB moves from one valid state to another
// Constraints, triggers, foreign keys enforced

// I — Isolation: concurrent transactions don't interfere
// Levels: READ UNCOMMITTED → READ COMMITTED → REPEATABLE READ → SERIALIZABLE

// D — Durability: committed data survives crashes (WAL, fsync)
Real-World
PostgreSQL, MySQL InnoDB, Oracle — any system where correctness beats throughput: banking, booking systems, medical records.

Consistency Models

Distributed ▼
Spectrum from linearizable (strongest) to eventual (weakest) — defines what readers see after writes.
Hierarchy (strongest → weakest)

//    Every op appears to happen atomically at one point in time

//    → Zookeeper

// 3. Causal Consistency
//    Causally related ops seen in order; concurrent ops may differ

//    You always see your own writes (others may lag)

// 5. Eventual Consistency (weakest)
//    Given enough time, all replicas converge
//    → DynamoDB, DNS, CDN caches

Paxos

Distributed ▼
Consensus protocol: proposers, acceptors, learners reach agreement in async networks with failures.
Phases
// Phase 1: Prepare
// Proposer sends Prepare(n) to majority of acceptors
// Acceptors: if n > any previously seen, promise not to accept lower
//            return any previously accepted value

// Phase 2: Accept
// If proposer gets majority promises:
//   - If any acceptor returned a value, propose that value (preserves safety)
//   - Otherwise, propose its own value
// Send Accept(n, value) to acceptors
// Acceptors: accept if no higher-numbered prepare seen

// Result: Once majority accepts → value is chosen → learners notified
Strengths
  • Proven correct (safety guaranteed)
  • Tolerates minority failures
Weaknesses
  • Notoriously hard to understand & implement
  • Livelock possible with competing proposers
  • Multi-Paxos needed for practical systems

CRDT

Distributed ▼
Conflict-free Replicated Data Types — merge without coordination, always converge.
How It Works
// G-Counter (grow-only counter) — simplest CRDT
// Each node maintains its own counter
// Merge: take max of each node's counter
Node A: {A: 3, B: 0}
Node B: {A: 1, B: 5}
Merge:  {A: 3, B: 5} → total = 8

// LWW-Register (Last-Writer-Wins): timestamp resolves conflicts
// OR-Set (Observed-Remove Set): add/remove without conflicts

// Key property: merge is commutative, associative, idempotent
// merge(A, B) = merge(B, A)           — order doesn't matter
// merge(A, merge(B, C)) = merge(merge(A, B), C)
// merge(A, A) = A                     — safe to re-merge
Real-World
Figma (multiplayer design), Redis CRDT (geo-distributed), Riak (distributed KV store), Yjs/Automerge (collaborative text editing).

Low-Level & Performance

CAS & Atomics

Low-Level ▼
Compare-and-swap: lock-free atomic update — if value == expected, swap it, else retry.
How It Works
// CAS is a single CPU instruction (CMPXCHG on x86)
// Pseudocode:
function compareAndSwap(address, expected, newValue) {
  if (*address === expected) {  // atomic check
    *address = newValue;        // atomic write
    return true;
  }
  return false;
}

// Lock-free counter increment:
do {
  old = counter.get();
} while (!counter.compareAndSet(old, old + 1));

// Java: AtomicInteger, AtomicReference
// C++: std::atomic, Rust: std::sync::atomic
// JS: Atomics.compareExchange (SharedArrayBuffer)
Strengths
  • No locks → no deadlocks
  • Better performance under low contention
Weaknesses
  • ABA problem (value changed and changed back)
  • Spin-loop waste under high contention

Memory Model

Low-Level ▼
Rules for how threads see each other's memory writes — defines visibility and ordering guarantees.
Problem
CPUs and compilers reorder instructions for performance. Thread A writes x=1; ready=true, but Thread B might see ready=true before x=1!
Ordering Levels
// Relaxed — no ordering guarantees (fastest)
// Acquire/Release — synchronized pairs (publish/subscribe pattern)
// Sequential Consistency — total order (slowest, safest)

// Java: volatile = acquire + release semantics
volatile boolean ready = false;
// Thread A: x = 42; ready = true;    // release
// Thread B: if (ready) use(x);       // acquire — guaranteed to see x=42

// Happens-before relationship:
// volatile write → volatile read (same variable)
// synchronized block exit → synchronized block entry (same monitor)
// Thread.start() → first action in new thread

Garbage Collection

Low-Level ▼
Automatic memory reclamation — trades manual control for safety, with pause-time trade-offs.
Strategies
// 1. Reference Counting (Python, Swift, Obj-C)
//    Each object tracks # of references. Freed when count = 0.
//    ✓ Deterministic, incremental    ✗ Circular references leak

// 2. Mark-and-Sweep (JS V8, Java, Go)
//    Phase 1: Mark all reachable objects from roots
//    Phase 2: Sweep (free) unmarked objects
//    ✓ Handles cycles    ✗ Stop-the-world pauses

// 3. Generational GC (V8, JVM HotSpot)
//    Young gen: small, collected often (most objects die young)
//    Old gen: large, collected rarely
//    ✓ Optimized for common patterns    ✗ Promotion overhead

// 4. No GC — manual (C) or ownership (Rust)
//    Rust: compiler tracks lifetimes, frees at scope exit
//    ✓ Zero overhead    ✗ Learning curve, borrow checker fights

Thread Safety

Concurrency ▼
Code that works correctly under concurrent access — via locks, immutability, or confinement.
Bad Approach
// Race condition — two threads incrementing shared counter
let count = 0;
// Thread A: temp = count (0); temp++; count = temp (1)
// Thread B: temp = count (0); temp++; count = temp (1)
// Expected: 2, Got: 1
Solutions
// 1. Mutual Exclusion (locks/mutexes)
mutex.lock();
count++;
mutex.unlock();

// 2. Atomic operations
atomicCount.incrementAndGet();

// 3. Immutability — no writes = no races
const config = Object.freeze({ port: 3000 });

// 4. Thread confinement — each thread owns its data
ThreadLocal<Connection> conn = ThreadLocal.withInitial(pool::getConn);

// 5. Message passing — share by communicating (Go channels, actors)
ch <- value // send
v := <-ch   // receive

Fault Tolerance

Reliability ▼
System continues operating despite component failures — circuit breakers, retries, redundancy.
Patterns
// 1. Circuit Breaker — stop calling a failing service
class CircuitBreaker {
  state = "CLOSED"; failures = 0; threshold = 5;

  async call(fn) {
    if (this.state === "OPEN") throw new Error("Circuit open");
    try {
      const result = await fn();
      this.failures = 0;
      return result;
    } catch (e) {
      if (++this.failures >= this.threshold) this.state = "OPEN";
      throw e;
    }
  }
}
